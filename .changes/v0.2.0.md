## [v0.2.0](https://github.com/irl-llc/git-spice-code-extension/releases/tag/vv0.2.0) - 2026-06-16
### Changed
- The CI change-fragment gate now treats `.claude/` (Claude Code configuration) and `.status-pipe/` (agent-loop protocol files) as non-release agent/dev tooling, alongside the existing `src/test/` and `scripts/` paths, so changes to that tooling no longer require a changelog fragment or a `skip-changelog` label.
### Fixed
- Scheduled auto-releases now actually publish to the VS Code Marketplace and Open VSX. Previously a release tagged by the scheduled job never published, because GitHub does not re-trigger workflows from a Release created with the default token; the auto-release run now chains the publish step itself via a shared reusable workflow.
- Fixed a refresh storm where a multi-step git operation (e.g. `gs stack submit`) triggered continuous stack-view refreshes for the whole command. Watcher-driven refreshes are now held while the operation's git markers (index lock, rebase/merge state) are present and fire once they clear, collapsing the burst into a single trailing refresh with no fixed delay.
- Fixed the stack view failing to load in a linked worktree of a bare repository (git "exit 128"): the extension now resolves the real working-tree root via `git rev-parse --show-toplevel` before running git-spice, and the file watcher follows the worktree's actual git-dir and shared common dir instead of assuming `.git` is a directory, so branch and HEAD changes still trigger refreshes.
